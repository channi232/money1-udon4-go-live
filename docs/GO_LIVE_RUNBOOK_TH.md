# Go-Live Runbook (Thai)

เอกสารนี้ใช้เป็น SOP สำหรับวันเปิดใช้งานจริงของระบบรวม `money.udon4.go.th` และใช้ร่วมกับหน้า `/go-live` และ `/readiness`

## 1) วัตถุประสงค์

- เปิดใช้งานระบบใหม่อย่างมีขั้นตอน ตรวจสอบได้ และ rollback ได้ทันทีเมื่อจำเป็น
- ลดผลกระทบต่อผู้ใช้งานจริงในช่วง 24 ชั่วโมงแรก

## 2) Go-Live Gate (ต้องผ่านก่อนเริ่ม)

1. Readiness ผ่าน 100% จากหน้า `/readiness`
2. Security baseline ผ่าน (headers, auth source, method enforcement)
3. ธุรกิจยืนยันความถูกต้องข้อมูลเทียบระบบเดิม (อย่างน้อย 20 ราย, 2 งวด)
4. Backup ล่าสุดพร้อมใช้งาน และทีมทดสอบ restore ขั้นต้นแล้ว

ถ้าไม่ผ่านข้อใดข้อหนึ่ง: **ไม่เปิดใช้งานจริง**

## 3) บทบาทผู้รับผิดชอบ

- System Admin: ดูแล deploy, monitoring, rollback
- Finance Lead: ยืนยันความถูกต้องฝั่งการเงิน
- Personnel Lead: ยืนยันข้อมูลบุคลากรและสิทธิ์ใช้งาน
- Incident Coordinator: สื่อสารสถานะกับผู้เกี่ยวข้อง

## 4) Timeline วันเปิดใช้งาน

### T-1 วัน

- Freeze โค้ด (งดแก้ไขที่ไม่จำเป็น)
- สำรอง `public_html`, config, logs และ snapshot ฐานข้อมูลตามนโยบาย
- ยืนยันรายชื่อผู้รับผิดชอบช่วงเฝ้าระวัง

### T-2 ชั่วโมง

- ตรวจ `/readiness` ให้ผ่านครบ
- ตรวจ `/api/session.php?debug=1` ว่า authenticated ถูกต้อง
- ตรวจหน้าหลัก 3 โมดูล: `/money`, `/slip`, `/tax`

### T-0 (เริ่ม Go-Live)

- ประกาศเริ่มใช้งานจริง
- เฝ้าระวัง `/security` และ `/audit` แบบใกล้ชิด

### T+2 ชั่วโมง

- สุ่มตรวจธุรกรรมจริงอย่างน้อย 10-20 รายการ
- บันทึกปัญหาและการแก้ไขทุกรายการ

### T+24 ชั่วโมง

- สรุปผลวันแรก
- ตัดสินใจคงระบบใหม่หรือดำเนินการ rollback ตามเงื่อนไข

## 5) เงื่อนไขตัดสินใจ Rollback

ดำเนินการ rollback ทันทีหากเกิดข้อใดข้อหนึ่ง:

- ผู้ใช้หลักเข้าใช้งานไม่ได้ต่อเนื่องเกิน 15 นาที
- พบความผิดพลาดข้อมูลเชิงธุรกิจที่กระทบการเงินจริง
- พบเหตุความปลอดภัยระดับวิกฤตที่ยังควบคุมไม่ได้

## 6) ขั้นตอน Rollback ฉุกเฉิน

1. ประกาศ incident และหยุดการเปลี่ยนแปลงทั้งหมด
2. Restore โค้ดจาก backup ล่าสุดที่ผ่านการตรวจ
3. ตรวจ endpoint สำคัญ (`/api/session.php`, `/readiness`, `/money`, `/slip`, `/tax`)
4. แจ้งผู้ใช้งานและผู้บริหารเรื่องสถานะ/เวลาแก้ไข
5. ทำ post-incident review ภายใน 24 ชั่วโมง

## 7) หลักฐานที่ต้องเก็บ

- Screenshot หน้า `/readiness` ก่อนเปิดใช้งาน
- รายการตรวจจาก `/go-live` พร้อมลายเซ็นผู้รับผิดชอบ
- Log เหตุการณ์จาก `/audit` และ `/security` ในช่วง T-0 ถึง T+24h
- สรุปผลวันแรกและข้อเสนอปรับปรุง
