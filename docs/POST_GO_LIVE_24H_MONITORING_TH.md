# Post Go-Live Monitoring (24 ชั่วโมงแรก)

เอกสารนี้เป็นเช็กลิสต์ปฏิบัติการหลังเปิดใช้งานจริง โดยเน้นการใช้งาน `request_id`, `error_code` และ `support trace` เพื่อให้ทีม support/ops จัดการเหตุได้เร็ว

## 1) เป้าหมายช่วง 24 ชั่วโมงแรก

- จับปัญหาให้เร็วที่สุดและจำกัดผลกระทบผู้ใช้
- ระบุสาเหตุเชิงเทคนิคได้จากหลักฐานเดียวกันทั้งทีม
- ตัดสินใจ hotfix/rollback อย่างมีข้อมูล

## 2) รอบตรวจที่แนะนำ

- `T+0 ถึง T+2h`: ตรวจทุก 15 นาที
- `T+2h ถึง T+8h`: ตรวจทุก 30 นาที
- `T+8h ถึง T+24h`: ตรวจทุก 60 นาที

## 3) สิ่งที่ต้องตรวจทุกครั้ง

1. หน้าโมดูล `money`, `tax`, `slip` เปิดได้ปกติ
2. แหล่งข้อมูลขึ้นเป็น `ฐานข้อมูลจริง` ตามคาด
3. กรณีผิดปกติให้คัดลอก `support trace` (admin) ทันที
4. ตรวจ error/fallback จาก API ว่ามี `error_code` หรือไม่
5. บันทึกเหตุใน incident log (เวลา, อาการ, trace, การแก้)

## 4) มาตรฐานการรับแจ้งเหตุ (Incident Intake)

เมื่อรับแจ้งปัญหา ให้เก็บข้อมูลขั้นต่ำดังนี้:

- โมดูลที่พบปัญหา (`money` | `tax` | `slip`)
- เวลาเกิดเหตุ (พร้อม timezone)
- `support trace` ที่คัดลอกจากหน้า (รูปแบบ `req=... | code=... | stage=...`)
- ภาพหน้าจออาการ
- บทบาทผู้ใช้ที่พบปัญหา (admin/finance/personnel)

## 5) แนวทางวิเคราะห์เร็วจาก error_code

- กลุ่ม `*_DB_CONFIG_*`: ตรวจไฟล์ config/สิทธิ์อ่านไฟล์บนเซิร์ฟเวอร์
- กลุ่ม `*_DB_UNAVAILABLE`: ตรวจสถานะ DB, network, timeout
- กลุ่ม `*_NO_TABLE_FOUND` หรือ `*_SCHEMA_*`: ตรวจ `schema-map.php` และชื่อ table จริง
- กลุ่ม `*_REQUIRED_COLUMNS_UNMAPPED`: ตรวจชื่อคอลัมน์จริง เทียบ schema-map
- กลุ่ม `*_QUERY_FAILED`: ตรวจ SQL/runtime error จาก debug trace

## 6) เกณฑ์แจ้งเตือนทันที

- fallback ต่อเนื่องในโมดูลเดิมเกิน 2 รอบตรวจติดกัน
- error_code เดิมเกิดซ้ำเกิน 5 ครั้งใน 1 ชั่วโมง
- ผู้ใช้หลักใช้งานไม่ได้เกิน 10 นาที

## 7) เกณฑ์ปิดเหตุ (Close Incident)

- ผู้ใช้ทดสอบซ้ำแล้วใช้งานได้
- ไม่มีเหตุซ้ำใน 2 รอบตรวจถัดไป
- มีบันทึก root cause + preventive action

## 8) สรุปส่งผู้บริหารเมื่อครบ 24 ชั่วโมง

- จำนวน incident ทั้งหมด
- MTTR เฉลี่ย
- top 3 สาเหตุที่พบบ่อย
- งานปรับปรุงที่ต้องทำต่อ (ลำดับความสำคัญ)
